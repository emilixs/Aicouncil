# LLM-Based Consensus Detection and Discussion Outcomes

**Issue:** ADM-12 — Improve consensus detection with LLM-based evaluation
**Date:** 2026-04-12
**Status:** Design

## Problem Statement

The current consensus detection in `council.service.ts` (lines 559-575) uses keyword matching against 8 hardcoded phrases ("i agree", "consensus reached", etc.). This has three critical flaws:

1. **False positives** — An expert saying "I agreed to take a different approach" or "We have consensus on the fact that we disagree" triggers consensus termination.
2. **False negatives** — Experts may express agreement using phrasing not in the keyword list (e.g., "This aligns perfectly with what everyone has proposed" or "I'm fully on board with this direction").
3. **Single-expert trigger** — Any one expert's message matching a keyword ends the discussion, regardless of whether other experts actually agree. There is no quorum or majority requirement.

Additionally, completed discussions produce no structured output — no summary, no action items, no record of what was decided. The only output is a boolean `consensusReached` flag and raw message history.

## Scope

This design covers four capabilities:

1. **LLM-based consensus evaluator** — Replace keyword matching with an LLM meta-evaluator
2. **Auto-generated discussion summary** — Produce a structured summary when discussion ends
3. **Voting/polling mechanism** — Allow explicit expert proposals and structured agreement
4. **Outcome tracking** — Store summaries, decisions, and action items in the database

## Design

### 1. LLM-Based Consensus Evaluator

#### Approach

Replace `detectConsensus(messageContent: string): boolean` with an async method that sends the full recent conversation to a dedicated LLM call (the "meta-evaluator"). This evaluator assesses convergence across all experts, not just the latest message.

#### When to evaluate

Run the consensus evaluator **after each complete round** (all experts have spoken once), not after every individual message. Rationale:
- Consensus is inherently a multi-party state; evaluating after one expert speaks is premature
- Reduces LLM API calls by `1/N` where N is expert count
- Gives every expert a chance to respond before assessment

Additionally, run after any round where an intervention was processed, since user input may redirect the discussion.

#### Meta-evaluator prompt

The evaluator receives:
- The problem statement
- List of participating experts and their specialties
- All messages from the current round (one per expert, the full text)
- For rounds 1-3: all prior messages verbatim. For rounds 4+: a condensed summary of rounds 1 through (current-1), generated by a quick LLM summarization pass. This keeps context within reasonable token limits while preserving full detail for the current round being evaluated.

It returns a structured JSON response:

```typescript
interface ConsensusEvaluation {
  // Overall convergence score: 0.0 (total disagreement) to 1.0 (full consensus)
  convergenceScore: number;
  // Has meaningful consensus been reached?
  consensusReached: boolean;
  // What do experts agree on?
  areasOfAgreement: string[];
  // What remains contested?
  areasOfDisagreement: string[];
  // Is the discussion still making progress or going in circles?
  progressAssessment: 'converging' | 'stalled' | 'diverging';
  // Brief reasoning for the assessment
  reasoning: string;
}
```

#### Consensus threshold

Consensus is reached when `convergenceScore >= 0.8` AND `consensusReached === true`. Both conditions required to prevent the LLM from gaming either field independently.

The threshold (0.8) should be configurable per session via a new `consensusThreshold` field on the Session model (default 0.8).

#### Which LLM to use

The meta-evaluator should use a capable model (e.g., Claude Sonnet or GPT-4o) regardless of which models the individual experts use. The evaluator needs strong reasoning to assess nuanced agreement. Configuration:

- New env var `CONSENSUS_EVALUATOR_MODEL` (default: the Anthropic driver with `claude-sonnet-4-20250514`)
- New env var `CONSENSUS_EVALUATOR_DRIVER` (default: `ANTHROPIC`)

The evaluator driver is created once and reused across sessions (unlike per-expert drivers).

#### Stall detection

If `progressAssessment === 'stalled'` for 2 consecutive rounds, emit a new event `discussion.stalled` and optionally inject a system message prompting experts to either find common ground or articulate remaining disagreements clearly. After 3 consecutive stalled rounds, auto-end the discussion with reason `stalled`.

#### Integration point

In `CouncilService.startDiscussion()`, replace the per-message `detectConsensus()` call (line 372) with a per-round evaluation. Move the check to after the round-increment logic (after line 420), so it runs once all experts in the round have spoken:

```typescript
// After incrementing currentRound (line 420):
if ((currentExpertIndex % experts.length === 0) && currentExpertIndex > 0) {
  const evaluation = await this.consensusService.evaluateConsensus(
    sessionId, session, experts, currentRound - 1
  );
  consensusReached = evaluation.consensusReached
    && evaluation.convergenceScore >= session.consensusThreshold;
}
```

The `currentRound - 1` is because `currentRound` was already incremented for the next round.

#### Remove keyword instruction from system prompt

The current system prompt (line 518) tells experts to say "I agree" or "consensus reached". This instruction should be removed. Instead, experts should be instructed to express their genuine position. The meta-evaluator handles detection externally.

New instruction:
```
You are participating in a collaborative discussion with other experts. Engage substantively with the problem and with other experts' positions. Express agreement or disagreement with specific points. Build on good ideas and challenge weak ones.
```

### 2. Auto-Generated Discussion Summary

#### When to generate

Generate a summary whenever a discussion ends, regardless of the reason (consensus, max_messages, stalled, cancelled). The summary is generated asynchronously after the session status is updated, so it doesn't block the session-ended event.

#### Summary structure

```typescript
interface DiscussionSummary {
  // One-paragraph executive summary
  executiveSummary: string;
  // Key decisions or recommendations
  decisions: string[];
  // Extracted action items with optional assignee hints
  actionItems: Array<{
    description: string;
    priority: 'high' | 'medium' | 'low';
    suggestedOwner?: string; // Expert name or role, not a system ID
  }>;
  // Key arguments that shaped the outcome
  keyArguments: Array<{
    expertName: string;
    position: string;
  }>;
  // Unresolved questions or disagreements
  openQuestions: string[];
  // The final consensus evaluation (if available)
  finalEvaluation?: ConsensusEvaluation;
}
```

#### Generation approach

Use the same meta-evaluator LLM (or a separate summary-optimized prompt). Feed it all messages (not just last 10), the problem statement, and the final consensus evaluation. The prompt asks for structured JSON output matching the `DiscussionSummary` interface.

For long discussions (>50 messages), use a two-pass approach:
1. Summarize messages in chunks of 20, producing intermediate summaries
2. Synthesize intermediate summaries into the final `DiscussionSummary`

#### Delivery

- Stored in a new `DiscussionOutcome` table (see Section 4)
- Emitted via a new WebSocket event `discussion-summary` to the session room
- Available via a new REST endpoint `GET /sessions/:id/outcome`

### 3. Voting/Polling Mechanism

#### Purpose

Allow structured, explicit agreement tracking alongside the LLM-based evaluator. This serves two use cases:
- Users can inject a poll to ask experts to vote on a specific proposal
- The system can automatically poll experts on the leading proposal when convergence is high but not yet definitive

#### Mechanism

A "poll" is a special system message injected into the discussion that asks each expert to respond with a structured vote on a specific proposal.

```typescript
interface Poll {
  id: string;
  sessionId: string;
  proposal: string;
  createdAt: Date;
  createdBy: 'user' | 'system'; // user intervention or auto-triggered
  status: 'open' | 'closed';
}

interface PollVote {
  id: string;
  pollId: string;
  expertId: string;
  vote: 'agree' | 'disagree' | 'agree_with_reservations';
  reasoning: string;
  createdAt: Date;
}
```

#### How voting works

1. A poll is created (by user intervention or auto-triggered when `convergenceScore >= 0.7`).
2. The poll proposal is injected as a SYSTEM message: `"POLL: Please respond to the following proposal with your vote (agree/disagree/agree_with_reservations) and brief reasoning: {proposal}"`.
3. Each expert's next turn response is parsed for the vote. The meta-evaluator extracts the structured vote from the expert's natural-language response (experts don't need to format their response specially).
4. After all experts have responded, the poll is closed and results are tallied.
5. Poll results feed into the next consensus evaluation as additional signal.

#### Auto-poll trigger

When the consensus evaluator returns `convergenceScore >= 0.7` but `consensusReached === false`, the system extracts the leading proposal from `areasOfAgreement` and automatically creates a poll. This happens at most once per session to avoid poll fatigue.

#### WebSocket events

- `poll-created` — `{ sessionId, pollId, proposal }`
- `poll-vote` — `{ sessionId, pollId, expertId, expertName, vote, reasoning }`
- `poll-closed` — `{ sessionId, pollId, results: { agree, disagree, agreeWithReservations } }`

#### REST endpoint

- `POST /sessions/:id/poll` — Create a poll (user-initiated)
- `GET /sessions/:id/polls` — List polls for a session

### 4. Outcome Tracking — Database Schema

#### New models

```prisma
model DiscussionOutcome {
  id               String   @id @default(cuid())
  sessionId        String   @unique
  executiveSummary String   @db.Text
  decisions        Json     // string[]
  actionItems      Json     // Array<{ description, priority, suggestedOwner? }>
  keyArguments     Json     // Array<{ expertName, position }>
  openQuestions    Json     // string[]
  finalEvaluation  Json?    // ConsensusEvaluation object
  generatedAt      DateTime @default(now())
  generatedBy      String?  // model name used for generation

  session Session @relation(fields: [sessionId], references: [id], onDelete: Cascade)
}

model ConsensusEvaluation {
  id                  String   @id @default(cuid())
  sessionId           String
  roundNumber         Int
  convergenceScore    Float
  consensusReached    Boolean
  areasOfAgreement    Json     // string[]
  areasOfDisagreement Json     // string[]
  progressAssessment  String   // 'converging' | 'stalled' | 'diverging'
  reasoning           String   @db.Text
  evaluatedAt         DateTime @default(now())

  session Session @relation(fields: [sessionId], references: [id], onDelete: Cascade)

  @@index([sessionId])
  @@index([roundNumber])
}

model Poll {
  id        String     @id @default(cuid())
  sessionId String
  proposal  String     @db.Text
  createdBy String     // 'user' | 'system'
  status    String     @default("open") // 'open' | 'closed'
  createdAt DateTime   @default(now())
  closedAt  DateTime?

  votes   PollVote[]
  session Session    @relation(fields: [sessionId], references: [id], onDelete: Cascade)

  @@index([sessionId])
}

model PollVote {
  id        String   @id @default(cuid())
  pollId    String
  expertId  String
  vote      String   // 'agree' | 'disagree' | 'agree_with_reservations'
  reasoning String   @db.Text
  createdAt DateTime @default(now())

  poll   Poll   @relation(fields: [pollId], references: [id], onDelete: Cascade)
  expert Expert @relation(fields: [expertId], references: [id], onDelete: Cascade)

  @@unique([pollId, expertId])
  @@index([pollId])
}
```

#### Session model changes

Add to the existing `Session` model:

```prisma
model Session {

```

#### Expert model changes

Add relation:

```prisma

  // ... existing fields/relations ...

}
```

### 5. New Service: ConsensusService

Extract consensus logic from `CouncilService` into a dedicated `ConsensusService`:

```
src/consensus/
  consensus.module.ts
  consensus.service.ts        — evaluateConsensus(), generateSummary()
  consensus.controller.ts     — REST endpoints for outcomes
  dto/
    consensus-evaluation.dto.ts
    discussion-outcome.dto.ts
    poll.dto.ts
  prompts/
    consensus-evaluator.prompt.ts   — meta-evaluator system prompt
    summary-generator.prompt.ts     — summary generation prompt
    vote-extractor.prompt.ts        — extract structured votes from expert responses
```

This keeps `CouncilService` focused on orchestration while `ConsensusService` owns evaluation logic.

## Key Design Decisions

| Decision | Rationale |
|----------|-----------|
| Evaluate per-round, not per-message | Consensus is multi-party; evaluating after one expert is premature. Reduces API costs. |
| Dedicated meta-evaluator model | Expert models vary in capability; consensus evaluation needs consistent quality. |
| Store every evaluation | Enables convergence visualization over time and debugging. |
| Auto-poll at 0.7 convergence | Bridges the gap between organic convergence and explicit agreement. Once per session to avoid fatigue. |
| Async summary generation | Don't block session-ended event; summary can take a few seconds. |
| Two-pass summarization for long discussions | Avoids context window limits while preserving full discussion content. |
| Remove "say I agree" instruction | Let experts argue naturally; external evaluation is more reliable than elicited keywords. |
| Stall detection with auto-end | Prevents infinite loops where experts talk past each other. |

## Out of Scope

- Real-time convergence visualization in the frontend (UI changes are a separate issue)
- Weighting expert votes by expertise relevance
- Multi-round polling (only single polls per session in v1)
- Streaming the summary generation to the client
- Customizable summary templates

## Migration Notes

- This is a **breaking change** for consensus behavior. Existing sessions with `consensusReached: true` were determined by keyword matching; future sessions will use LLM evaluation.
- The `detectConsensus()` keyword method should be removed entirely, not kept as fallback. Keeping it creates ambiguity about which system is authoritative.
- Existing `Session` records need no backfill — `consensusThreshold` defaults to 0.8 and new relations are optional.

## Testing Strategy

- Unit tests for `ConsensusService` with mocked LLM responses (structured JSON parsing, threshold logic, stall detection)
- Integration test: run a short discussion (3 experts, 2 rounds) and verify consensus evaluation is stored and summary is generated
- Edge cases: empty discussions, single-expert sessions, LLM returning malformed JSON, convergence score exactly at threshold
- Poll flow: create poll, simulate expert responses, verify vote extraction and tally
